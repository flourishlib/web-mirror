I don't have a set date for reaching stable state, but I can elaborate on what I've been thinking for making Flourish stable.

Right now as being in beta, I'm committed to making the absolute minimum number of backwards compatibility breaks as possible. Since Flourish became beta in November there have been around 100 new revision and only 2 were backwards incompatible. For the most part the Flourish API is stable.

However, the project being stable is not just the API but also how buggy it is. Flourish is in beta to try and flush out bugs through real-world usage. So far I've gotten a good number of tickets about MySQL functionality, which isn't a surprise since I primarily use PostgreSQL at work and on personal projects.

Beyond that, I would like to get to a point where there is test coverage on the majority of code/functionality in the library. As it stands I've been adding new tests as I fix bugs and writing others to try and reinforce things.

The real-world usage of Flourish has definitely been beneficial. It has been in production use since even before it became beta and has been used on over twenty sites by myself and co-workers. One last thing to note is that I try to make sure all bugs are fixed within 24-48 hours so that end-user development is not held up.
