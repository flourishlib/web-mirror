PHP already has a fully featured XML parser extension in the base package, installed by default, and it's pretty powerful as-is given that XML is a very simple data format. Where compiled, built-in functions exist, I feel like flourish should refrain from recreating them in PHP, a slower interpreted language. This is different than collecting common use scenarios into single classes of course, and I wouldn't put it past Will to come up with some way of making PHP's XML parser even easier/better inside fXML.

As for documentation, Will seems pretty good about expanding documentation if you start a ticket for those areas you feel are unclear. Without anything specific, there's not much to do. Is this a style critique? :)

You're right, fSession is pretty simple and one probably could write it oneself if one were at all competent in PHP, but it solves important code separation and reuse issues for the flourish library. There are other flourish classes which use sessions in order to provide their respective functions (fMessaging, for instance), and without fSession they'd be repeating code needlessly and messily all over the place.

As for classes using a prefix, I'm not sure if you mean flourish should expand on the lowercase "f" or if they should have a prefix that identifies them as classes. In the case of the latter, I'd wonder if that would be overkill as flourish is entirely class-based.

The best thing about flourish is that for everything it does, you can still easily augment it with other libraries and classes. I've grown quite fond of using flourish with a few of the micro-frameworks that have appeared lately, for example. But simply looking at the backwards compatibility breaks will tell you flourish is very much beta, and growing quite fast :)

posted by wbond 10 years ago

Hi Jay, thanks for taking the time to look at Flourish.

It is true that some of the code in Flourish is not focused on providing tons of functionality, but instead of creating readable code. I know off of the top of my head that there are some single line methods in fHTML, fXML and fRequest. These usually are much more clear when reading code as opposed to calling nested function calls or having to include 3 or more parameters to ensure that everything uses UTF-8.

If you do have some specific feedback on documentation that isn't clear, or seems lacking, feel free to submit a ticket or start a forum post about it. I know that using third-party code is hard when there aren't enough examples or there is no documentation.

Vena was correct in reference to fSession. fSession is primarily a consistent interface for other Flourish code, but also have the benefit of namespacing session values and allow for default values in a call to fSession::get() instead of having to use the ternary operator with isset(). It also happens to include some good security features when starting the session.

I don't currently have plans to extends fXML much since I find that SimpleXML does a pretty good job. That's not to say I love SimpleXML, but I also don't work with XML a ton. I prefer to use JSON where possible.

In regards to forms, with Flourish I do try to stay away from generating HTML as much as possible. Many of the MVC frameworks provide lots of HTML helpers and stuff, but it is personally not a path I am interested in heading down. You will find that using fActiveRecord as an ORM helps cut down on validation of forms since the record is validated against the database structure, with additional options provided by fORMValidation. For simple, non-database-backed form, fValidation provides some simple tools.
